# Spec — 내부결재시스템

> **버전**: 1.0  
> **작성일**: 2026-04-02  
> **상태**: Draft

---

## 1. Problem Statement

행정기관 내부의 종이 기반 결재 프로세스는 다음과 같은 문제를 야기한다:
- **시간 지연**: 물리적 문서 이동으로 결재까지 수시간~수일 소요
- **위치 제약**: 결재자가 책상에 없으면 결재 진행 불가
- **이력 추적困难**: 종이 문서의 분실/훼손 시 이력 복원 불가
- **비용**: 용지, 인쇄, 보관 공간 유지 비용 발생
- **비효율**: 반복적 결재 양식 수기 작성, 결재선 수동 지정

## 2. Business Value

- **결재 시간 70% 단축**: 전자적 자동 라우팅으로 실시간 결재 진행
- **공간/비용 절감**: 종이 문서 보관 공간 및 인쇄 비용 제로화
- **투명성 확보**: 결재 전 과정의 디지털 감사추적 가능
- **모바일/원격 결재**: 시간/장소 제약 없는 결재 처리
- **법적 준수**: 행정 전자결재 기준 준수, 감사 대응 용이

## 3. User Scenarios & Testing

### US1 [P1] — 기안작성: 직원이 결재 문서를 작성하여 제출한다

**배경**: 직원이 업무 관련 결재가 필요할 때 전자 문서를 작성하여 결재선에 제출한다.  
**액터**: 일반 직원 (DRAFTER)  
**전제조건**: 로그인 상태, 결재 양식 템플릿 존재  

**시나리오**:
1. 직원이 결재 양식 목록에서 원하는 양식을 선택한다
2. 필수 항목 (제목, 내용, 참고사항, 첨부파일) 을 입력한다
3. 결재선을 선택하거나 기존 결재선 템플릿을 적용한다
4. 임시저장하거나 제출한다
5. 제출 시 결재선 1순위 결재자에게 알림이 전달된다

**독립 테스트**: 기안 문서 생성 API 호출 → DB 저장 확인 → 결재 상태 "진행중" 확인

---

### US2 [P1] — 결재선지정: 관리자가 결재 라인(결재자 순서)을 설정한다

**배경**: 결재 문서에 적용될 결재자 순서를 정의한다. 결재선은 부서/문서종류별로 템플릿화할 수 있다.  
**액터**: 부서장 또는 시스템 관리자 (ORG_ADMIN)  
**전제조건**: 인사DB 연동 완료, 조직도 정보 존재  

**시나리오**:
1. 관리자가 결재선 설정 화면에서 부서/문서종류를 선택한다
2. 인사DB에서 결재자 후보 목록을 조회한다
3. 결재 순서 (1순위, 2순위, ... 최종결재자) 를 드래그앤드롭으로 지정한다
4. 결재선을 저장하면 템플릿으로 재사용 가능하다
5. 결재선 변경 시 기존 진행중 문서에는 영향이 없다 (스냅샷 방식)

**독립 테스트**: 결재선 생성 API → DB 저장 확인 → 결재선 조회 API로 순서 검증

---

### US3 [P1] — 승인반려: 결재자가 문서를 승인하거나 반려한다

**배행**: 결재자가 자신에게 할당된 결재 문서를 검토 후 승인 또는 반려 처리한다.  
**액터**: 결재자 (APPROVER)  
**전제조건**: 결재 문서가 자신의 순서에 도달함  

**시나리오**:
1. 결재자가 결재함에서 대기 중인 문서를 확인한다
2. 문서 내용, 첨부파일, 이전 결재자 의견을 검토한다
3. 승인 시: 의견을 작성하고 승인 처리 → 다음 결재자로 이관
4. 반려 시: 반려 사유를 필수 입력 → 기안자에게 반환
5. 최종결재자 승인 시 문서 상태 "완료"로 전환
6. 모든 결재 처리 내역은 감사로그에 기록된다

**독립 테스트**: 승인 API 호출 → 문서 상태 변경 확인 → 다음 결재자 할당 확인  
**독립 테스트**: 반려 API 호출 → 문서 상태 "반려" 확인 → 기안자에게 알림 확인

---

### US4 [P2] — 이력조회: 사용자가 결재 이력을 조회하고 검색한다

**배경**: 사용자는 자신이 관련있는 결재 문서의 이력을 조회하고 조건별 검색이 가능하다.  
**액터**: 모든 인증 사용자  
**전제조건**: 로그인 상태  

**시나리오**:
1. 사용자는 결재함 (내가 기안, 내가 결재, 참조문서) 을 확인한다
2. 문서종류, 기간, 상태, 결재자 등 조건으로 검색한다
3. 문서 상세 조회 시 결재 라인별 처리 현황 (승인/반려/대기) 을 시각화하여 확인한다
4. 결재 이력 타임라인을 통해 누가 언제 어떤 처리를 했는지 확인한다
5. 완료된 문서는 PDF로 출력/다운로드 가능하다

**독립 테스트**: 이력조회 API 호출 → 필터링 결과 검증 → 결재 타임라인 데이터 정확성 확인

---

### US5 [P2] — 인사DB 연동: 조직도 및 결재자 정보를 인사DB에서 동기화한다

**배경**: 결재선 지정 및 결재자 할당을 위해 인사DB의 조직/직원 정보를 주기적으로 동기화한다.  
**액터**: 시스템 (자동)  
**전제조건**: Oracle 19c 인사DB 연결 설정 완료  

**시나리오**:
1. 스케줄러가 매일 새벽 2시에 인사DB에서 조직도/직원 정보를 조회한다
2. 변경 사항 (입사, 퇴사, 부서이동, 직급변경) 을 감지하여 로컬 DB에 반영한다
3. 동기화 실패 시 관리자에게 알림 발송
4. 동기화 이력을 로그로 기록한다

**독립 테스트**: 동기화 잡 실행 → 로컬 DB 반영 확인 → 변경 이력 로그 확인

---

### US6 [P3] — 결재 문서 템플릿 관리: 자주 사용하는 결재 양식을 템플릿으로 관리한다

**배경**: 반복적으로 사용되는 결재 양식 (출장신청, 품의서, 지출결의서 등) 을 템플릿으로 등록하여 재사용한다.  
**액터**: 시스템 관리자 (SYSTEM_ADMIN)  

**시나리오**:
1. 관리자가 결재 양식 템플릿을 등록/수정/삭제한다
2. 템플릿은 HTML 기반 양식 구조와 필수 입력 필드를 정의한다
3. 사용자는 기안 시 템플릿을 선택하여 빠른 작성이 가능하다

**독립 테스트**: 템플릿 CRUD API → 양식 렌더링 확인 → 기안 시 템플릿 적용 확인

## 4. Edge Cases

| ID | 시나리오 | 처리 방안 |
|----|----------|-----------|
| E1 | 결재자가 장기 휴가/외출 중 | 결재선 지정 시 대체결재자 지정 가능 |
| E2 | 결재 처리 중 시스템 장애 | 트랜잭션 롤백, 재시도 시 동일 상태 유지 (멱등성) |
| E3 | 동시에 여러 결재자가 동일 문서 처리 | 비관적 락 (SELECT FOR UPDATE)으로 직렬화 |
| E4 | 결재선 중간에 퇴사자 포함 | 동기화 시 퇴사자 감지 → 관리자에게 결재선 수정 알림 |
| E5 | 반려 후 재제출 | 기존 문서 이력 유지, 새 버전으로 재제출 |
| E6 | 첨부파일 용량 초과 | 파일 크기 제한 (50MB), 초과 시 에러 메시지 |
| E7 | 결재선 순환 참조 (A→B→A) | 결재선 저장 시 순환 참조 검증, 발견 시 저장 거부 |
| E8 | 인사DB 연결 끊김 | 마지막 동기화 데이터로 운영, 연결 복구 후 자동 재동기화 |

## 7. Clarifications

### Session 2026-04-02

- Q: 첨부파일 저장 방식은? (로컬 파일시스템 vs DB BLOB vs NAS)
  - A: **Option A — 로컬 파일시스템 + DB 메타정보**. 폐쇄망에서 백업/복구가 단순하고, 파일 메타정보만 DB에 관리. 저장 경로는 설정 가능.

- Q: 결재선 지정 방식은? (기안자 직접 지정 vs 템플릿 기반 vs 관리자 지정)
  - A: **Option B — 기안자가 직접 지정**. 기안자가 결재자, 참조자를 직접 선택. 결재선 템플릿은 향후 확장으로预留.

- Q: 동시 사용자 수 및 성능 목표는?
  - A: 동시 사용자 약 **200명** (전사 1,000명 중). 페이지 응답 **2초 이내**. 대용량 기안문 **100KB 이내**.

- Q: 인증 방식은? (JWT vs 세션 vs 사내 인증서)
  - A: **Option A — 사번/사원번호 기반 인증**. 인사DB 연동 인증. 폐쇄망이므로 사내 인증서 기반.

- Q: 결재 상태 전이는 어떻게 되는가?
  - A: **임시저장 → 기안 → 상신 → 결재진행 → 승인** (완료). 반려 시 **반려 → 반려효수** (기안자가 수정 후 재상신). **취소**는 기안자가 상신 전까지 가능.

- Q: 결재 라인 구조는? (단일 vs 다단계)
  - A: **단일 결재선** (결재자 1인 → 참조자 N명). 결재자는 순차적. 향후 다단계 확장은 고려하지만 현재는 단일.

- Q: 검색/필터 조건은?
  - A: **결재 상태, 기안자, 기간(시작일~종료일), 제목 키워드**.

- Q: 기타 전제사항
  - A: UI는 **REST API만 제공** (프론트엔드 별도). 로그는 **DB 저장** (감사로그) + 운영서버 로컬 파일 로그. 결재문서 내용 **암호화 불필요**, 접근제어만. 세션 타임아웃: **30분**.
