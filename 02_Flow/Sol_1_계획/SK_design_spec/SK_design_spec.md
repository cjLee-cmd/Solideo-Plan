# SK_design_spec

## 목적

Speckit을 활용하여 신규 소프트웨어의 설계 문서(design.md)를 생성한다.
사용자로부터 프로젝트 요구사항을 수집하고, 체계적인 설계 문서를 산출한다.

## 입력 파라미터

| 파라미터 | 필수 | 설명 |
|----------|------|------|
| project_name | Y | 프로젝트명 |
| purpose | Y | 프로젝트 목적 및 배경 |
| requirements | Y | 기능 요구사항 목록 |
| tech_constraints | N | 기술스택 제약조건 (언어, 프레임워크, DB 등) |
| target_env | N | 배포 환경 (기본값: 폐쇄망) |
| existing_systems | N | 연동 대상 기존 시스템 목록 |

## 실행 프롬프트

```
당신은 행정망 소프트웨어 아키텍트입니다.
다음 요구사항을 기반으로 소프트웨어 설계 문서(design.md)를 작성하십시오.

[프로젝트 정보]
- 프로젝트명: {project_name}
- 목적: {purpose}
- 기능 요구사항: {requirements}
- 기술 제약조건: {tech_constraints}
- 배포 환경: {target_env}
- 연동 시스템: {existing_systems}

[설계 문서 구성]
아래 섹션을 모두 포함하여 design.md를 작성하십시오.

1. 프로젝트 개요
   - 프로젝트명, 목적, 범위, 용어 정의

2. 아키텍처 설계
   - 시스템 구성도 (텍스트 다이어그램)
   - 계층 구조 (Presentation / Business / Data)
   - 통신 방식 및 프로토콜

3. 모듈 구조
   - 모듈 목록 및 역할
   - 모듈 간 의존성
   - 디렉토리 구조

4. API 설계
   - 엔드포인트 목록 (메서드, 경로, 설명)
   - 요청/응답 스키마
   - 인증/인가 방식

5. DB 스키마
   - 테이블 목록 및 컬럼 정의
   - 관계 정의 (ERD 텍스트)
   - 인덱스 전략

6. 기술스택 결정
   - 언어, 프레임워크, 라이브러리
   - 선정 사유
   - 폐쇄망 호환성 확인

7. 보안 설계
   - 인증/인가 방식
   - 데이터 암호화
   - 감사 로그 정책

8. 비기능 요구사항
   - 성능 목표
   - 가용성
   - 확장성

[작성 규칙]
- 폐쇄망 환경을 전제로 외부 네트워크 의존 요소를 배제할 것
- 행정망 보안 지침을 준수할 것
- 모든 외부 라이브러리는 폐쇄망 내부 저장소에서 제공 가능한 것만 선정할 것
- 마크다운 형식으로 작성할 것
```

## 출력

| 산출물 | 형식 | 설명 |
|--------|------|------|
| design.md | Markdown | 전체 설계 문서 |

## 후속 연계

- SK_design_review: 생성된 design.md를 사용자 검증에 전달
- SW_planning: 계획 Workflow의 첫 번째 단계로 실행
