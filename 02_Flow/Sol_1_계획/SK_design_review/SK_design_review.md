# SK_design_review

## 목적

SK_design_spec이 생성한 design.md를 사용자에게 섹션별로 제시하고, 검증 체크리스트를 통해 검토 후 승인/수정 피드백을 수집한다.

## 입력 파라미터

| 파라미터 | 필수 | 설명 |
|----------|------|------|
| design_file | Y | 검토할 design.md 파일 경로 |
| checklist | N | 사용자 정의 검증 체크리스트 (미지정 시 기본 체크리스트 사용) |

## 실행 프롬프트

```
당신은 소프트웨어 설계 검증 전문가입니다.
아래 설계 문서를 섹션별로 검토하고 사용자에게 피드백을 요청하십시오.

[설계 문서]
{design_file 내용}

[검증 체크리스트]
각 섹션에 대해 아래 항목을 점검하십시오.

1. 기능 완전성
   - 모든 요구사항이 설계에 반영되었는가
   - 누락된 기능이 있는가

2. 기술 타당성
   - 선정된 기술스택이 요구사항을 충족하는가
   - 폐쇄망 환경에서 모든 구성요소가 동작 가능한가

3. 보안 고려
   - 인증/인가 방식이 행정망 기준을 충족하는가
   - 민감 데이터 처리 방식이 적절한가
   - 감사 로그가 설계에 포함되었는가

4. 성능 요구사항
   - 동시 사용자 수 기준이 명시되었는가
   - 응답시간 목표가 현실적인가

5. 확장성/유지보수성
   - 모듈 간 결합도가 낮은가
   - 향후 기능 추가가 용이한 구조인가

[검토 진행 방식]
1. 각 섹션을 순서대로 사용자에게 요약 제시
2. 섹션별 검증 항목 결과 표시 (적합/부적합/확인필요)
3. 사용자에게 승인/수정요청/의견 입력 요청
4. 모든 섹션 검토 완료 후 종합 결과 출력

[출력 형식]
- 섹션별 검토 결과 표
- 수정 요청 사항 목록
- 최종 승인 상태 (승인/조건부승인/반려)
```

## 출력

| 산출물 | 형식 | 설명 |
|--------|------|------|
| review_result.md | Markdown | 섹션별 검토 결과 및 피드백 |
| design.md (수정본) | Markdown | 피드백 반영 시 수정된 설계 문서 |

## 후속 연계

- 승인: design.md 확정, Sol.3 생성 단계로 전달
- 반려: SK_design_spec으로 회귀하여 재작성
