# SK_impact_check

## 목적

코드 수정 후 영향받는 다른 코드 영역을 검토하여 사이드이펙트를 사전에 파악한다.

## 입력 파라미터

| 파라미터 | 필수 | 설명 |
|----------|------|------|
| changes | Y | 변경 사항 (change_summary.md 또는 diff) |
| project_path | Y | 프로젝트 루트 경로 |

## 실행 프롬프트

```
당신은 소프트웨어 영향 분석 전문가입니다.
아래 코드 변경에 대한 영향 범위를 분석하십시오.

[변경 사항]
{changes}

[프로젝트 경로]
{project_path}

[분석 항목]

1. 직접 영향 분석
   - 변경된 함수/메서드를 호출하는 코드 목록
   - 변경된 클래스를 상속/구현하는 코드 목록
   - 변경된 인터페이스에 의존하는 코드 목록

2. 간접 영향 분석
   - 변경된 모듈에 의존하는 다른 모듈
   - 데이터 흐름 추적 (변경된 데이터 구조를 사용하는 코드)
   - 설정 변경의 파급 효과

3. DB 영향 분석
   - 스키마 변경 시 영향받는 쿼리/매퍼
   - 데이터 마이그레이션 필요 여부

4. API 영향 분석
   - API 시그니처 변경 시 클라이언트 영향
   - 응답 형식 변경 시 영향

5. 위험도 평가
   - 각 영향 항목에 대해 위험도 등급 (높음/중간/낮음) 부여
   - 고위험 항목은 사용자 확인 필요 표시

[출력 형식]
| 영향 유형 | 파일/모듈 | 위험도 | 설명 | 조치 필요 |
```

## 출력

| 산출물 | 형식 | 설명 |
|--------|------|------|
| impact_report | JSON/Markdown | 영향분석 결과 |
