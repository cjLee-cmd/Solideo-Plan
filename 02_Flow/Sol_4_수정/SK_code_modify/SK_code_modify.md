# SK_code_modify

## 목적

기존 소스코드를 버그 수정, 기능 추가, 리팩토링 등의 요구사항에 따라 수정한다. verify.md를 검증 기준으로 사용한다.

## 입력 파라미터

| 파라미터 | 필수 | 설명 |
|----------|------|------|
| target_path | Y | 수정 대상 파일/디렉토리 경로 |
| requirement | Y | 수정 요구사항 (텍스트 또는 파일 경로) |
| modify_type | N | 수정 유형 (bugfix/feature/refactor). 기본: auto |
| verify_file | N | 검증 기준 문서(verify.md) 경로 |

## 실행 프롬프트

```
당신은 행정망 소프트웨어 유지보수 전문가입니다.
아래 요구사항에 따라 기존 소스코드를 수정하십시오.

[수정 대상]
경로: {target_path}

[수정 요구사항]
{requirement}

[수정 유형]
{modify_type}

[검증 기준]
{verify_file 내용}

[수정 규칙]

1. 최소 변경 원칙
   - 요구사항에 필요한 최소 범위만 수정한다
   - 불필요한 리팩토링을 동시에 하지 않는다
   - 기존 코드 스타일을 유지한다

2. 변경 추적
   - 수정한 모든 파일의 변경 전/후를 기록한다
   - 변경 이유를 주석으로 남긴다
   - diff 형식으로 변경 사항을 정리한다

3. 하위 호환성
   - 기존 API 시그니처를 변경하지 않는다 (불가피한 경우 사유 명시)
   - 기존 데이터 형식과의 호환성을 유지한다
   - DB 스키마 변경 시 마이그레이션 스크립트를 포함한다

4. 보안 유지
   - 수정으로 인해 기존 보안 로직이 우회되지 않도록 한다
   - 새로 추가되는 입력은 검증 로직을 포함한다

[출력]
- 수정된 소스코드
- 변경 사항 요약 (diff)
- 수정 사유 설명
```

## 출력

| 산출물 | 형식 | 설명 |
|--------|------|------|
| 수정된 소스코드 | 각 언어별 | 수정 완료된 파일들 |
| change_summary.md | Markdown | 변경 사항 요약 (diff 포함) |
